## 2. 传统软件过程模型分析

### 2.1 瀑布模型

瀑布模型按照需求分析、概要设计、详细设计、编码、测试、部署和维护顺序推进，每个阶段有明确产出和评审节点。

| 维度 | 分析 |
|---|---|
| 优点 | 阶段清晰，文档完整，便于管理和验收。 |
| 缺点 | 对需求变化响应慢，用户较晚看到可运行结果，后期返工成本高。 |
| 适用场景 | 需求稳定、合同约束强、过程审查严格的项目。 |
| 对本项目的适用性 | 可借鉴其文档和阶段评审思想，但不宜完全采用。 |

人才招聘系统虽然角色和主流程较稳定，但界面细节、字段设计、审核逻辑和数据存储方式可能随实现过程调整。如果完全采用瀑布模型，容易在后期发现交互或数据设计问题，因此不作为主过程。

### 2.2 原型模型

原型模型先快速构建可操作原型，通过用户反馈逐步修正需求和设计。

| 维度 | 分析 |
|---|---|
| 优点 | 能尽早暴露需求理解偏差，适合界面和交互较多的系统。 |
| 缺点 | 若原型直接演化为正式系统，可能产生结构不清晰、质量不足的问题。 |
| 适用场景 | 用户对需求表达不明确、界面体验重要的项目。 |
| 对本项目的适用性 | 适合用于登录、注册、岗位管理和审核界面验证。 |

人才招聘系统是典型管理类 GUI 系统，原型能帮助确认表格字段、按钮流程、角色入口和错误提示，因此应吸收原型模型思想。

### 2.3 螺旋模型

螺旋模型以风险分析为核心，每轮迭代都经历目标确定、风险识别、开发验证和评审计划。

| 维度 | 分析 |
|---|---|
| 优点 | 重视风险管理，适合复杂或高风险项目。 |
| 缺点 | 管理成本高，对风险识别能力要求高。 |
| 适用场景 | 大型系统、技术风险高、需求不确定性大的项目。 |
| 对本项目的适用性 | 不适合作为完整过程，但适合用于风险登记和阶段复盘。 |

本项目规模较小，完整螺旋模型偏重。但数据安全、权限控制、进度延期等风险仍需要分级管理，因此可借鉴其风险驱动思想。

### 2.4 增量模型

增量模型将系统划分为多个功能增量，每次交付可运行的一部分功能。

| 维度 | 分析 |
|---|---|
| 优点 | 能较早交付可运行版本，降低一次性交付风险。 |
| 缺点 | 需要提前规划模块边界和公共架构。 |
| 适用场景 | 功能可以拆分、需要阶段性演示的项目。 |
| 对本项目的适用性 | 很适合按角色和业务链路分阶段实现。 |

人才招聘系统可以拆分为用户基础能力、岗位简历流程、审核留言流程、测试和文档完善等增量，适合采用增量式迭代。

## 3. 敏捷开发方法分析

### 3.1 敏捷开发总体特点

敏捷开发强调小步迭代、持续反馈、快速交付和拥抱变化。与传统模型相比，敏捷不否定文档和计划，但更强调可运行软件、客户协作和及时调整。

| 对比维度 | 传统过程模型 | 敏捷开发 |
|---|---|---|
| 需求处理 | 倾向前期冻结需求 | 允许需求逐步澄清和调整 |
| 交付节奏 | 阶段性交付 | 频繁交付可运行增量 |
| 文档 | 文档较重 | 保留必要文档，强调可用性 |
| 反馈 | 多在评审或测试阶段集中反馈 | 每轮迭代都收集反馈 |
| 管理方式 | 计划驱动 | 反馈驱动和价值驱动 |
| 风险暴露 | 后期风险可能集中爆发 | 通过短周期迭代提前暴露 |

### 3.2 Scrum

Scrum 是常见的敏捷框架，以固定周期 Sprint 推进工作，并通过透明、检视和适应来持续改进。

Scrum 的核心包括：

1. 角色：Product Owner、Scrum Master、Developers。
2. 事件：Sprint、Sprint Planning、Daily Scrum、Sprint Review、Sprint Retrospective。
3. 工件：Product Backlog、Sprint Backlog、Increment。

Scrum 适合本项目的原因：

1. 项目功能可拆分为多个可运行增量。
2. 每个实验和每周进度都可以作为检视点。
3. GitHub Projects 可以承载 Backlog 和任务状态。
4. 课程项目需要不断补充文档，Scrum 的 Review 和 Retrospective 有利于同步文档与代码。

### 3.3 XP

XP（Extreme Programming）强调工程实践，如测试驱动开发、持续集成、重构、结对编程和小版本发布。

| 实践 | 对本项目的应用 |
|---|---|
| 小版本发布 | 每完成一类角色或业务流程即形成可运行版本。 |
| 持续集成 | 每次提交前至少保证能构建或完成基本检查。 |
| 重构 | 当界面逻辑膨胀时，将业务规则迁移到 `RecruitmentService`。 |
| 测试优先思想 | 为注册、登录、审核、岗位申请等主流程建立回归清单。 |

本项目不需要完整实施 XP，但可以吸收其质量控制和持续验证思想。

### 3.4 Kanban

Kanban 通过看板可视化任务流，常见列包括 Todo、In Progress、Review、Done。

本项目可在 GitHub Projects 中使用 Kanban 风格进行任务跟踪：

1. Todo：未开始需求、文档或缺陷。
2. In Progress：正在完成的实验或功能。
3. Review：等待审阅确认的实验成果。
4. Done：已提交并推送的成果。

Kanban 适合日常任务跟踪，但课程阶段性成果较明确，因此更适合作为 Scrum 的辅助工具。

## 4. 本项目过程模型选择

### 4.1 项目特点

人才招聘系统具有以下特点：

1. 角色清晰：个人用户、企业用户、管理员。
2. 主流程稳定：注册审核、岗位发布、岗位查询、岗位申请、留言处理。
3. 界面细节易变化：表格字段、按钮布局、提示信息需要根据实现调整。
4. 项目周期有限，需要阶段性提交实验文档。
5. 代码和文档需要同步保存在 GitHub。

### 4.2 推荐过程

本项目推荐采用：

`适度前期设计 + Scrum 迭代开发 + Kanban 任务跟踪 + 局部 XP 工程实践`

### 4.3 选择理由

1. 适度前期设计保证角色、数据结构和核心流程不混乱。
2. Scrum 保证每个 Sprint 都有明确目标和可运行增量。
3. Kanban 让任务状态透明，便于审阅和进度跟踪。
4. XP 的测试、重构和持续验证思想能降低演示失败风险。
5. 该混合过程比纯瀑布更灵活，比完全无文档敏捷更适合课程提交。

### 4.4 过程落地方案

| 阶段 | 过程方式 | 主要工作 | 产出 |
|---|---|---|---|
| Sprint 0 | 前期设计 | 可行性分析、工具选型、需求范围、初始 Backlog | 可行性报告、CASE 调研、项目跟踪表 |
| Sprint 1 | Scrum 迭代 | 注册登录、用户审核、资料维护 | 可运行基础版本 |
| Sprint 2 | Scrum 迭代 | 岗位管理、岗位查询、岗位申请和撤销 | 招聘主流程版本 |
| Sprint 3 | Scrum 迭代 | 留言处理、权限校验、异常输入处理 | 管理闭环版本 |
| Sprint 4 | XP/测试强化 | 回归测试、重构、文档同步 | 演示版本和最终文档 |

## 5. Scrum 工作模型理解

### 5.1 Scrum 角色

| 角色 | 职责 | 本项目对应 |
|---|---|---|
| Product Owner | 明确产品目标、维护 Product Backlog、决定优先级 | 从用户价值角度排序功能：先主流程，后扩展功能。 |
| Scrum Master | 维护 Scrum 节奏、移除障碍、促进复盘 | 维护实验进度、风险表和 GitHub 任务状态。 |
| Developers | 完成分析、设计、编码、测试和文档 | 完成 Qt 代码、实验报告、模型和测试记录。 |

在课程项目中，角色可以由同一人兼任，但职责应区分。

### 5.2 Scrum 事件

| 事件 | 含义 | 本项目执行方式 |
|---|---|---|
| Sprint | 固定周期迭代容器 | 以一周或一个实验作为一个短周期。 |
| Sprint Planning | 确定本轮目标和任务 | 每次开始新实验前确定交付文件和核心功能。 |
| Daily Scrum | 检查进展并调整计划 | 用项目跟踪表记录当前任务、问题和下一步。 |
| Sprint Review | 展示增量并收集反馈 | 每完成一个实验交给审阅，通过后提交推送。 |
| Sprint Retrospective | 复盘过程并改进 | 每个实验结束后更新风险和跟踪表。 |

### 5.3 Scrum 工件

| 工件 | 含义 | 本项目产物 |
|---|---|---|
| Product Backlog | 项目需求总清单 | `scrum_backlog.md` 中的功能和文档任务。 |
| Sprint Backlog | 当前 Sprint 任务清单 | 每个实验目录下的交付文件和任务表。 |
| Increment | 可交付增量 | 已审阅并推送的实验文档或可运行代码。 |

### 5.4 完成定义

每个实验或功能的完成定义如下：

1. 满足对应实验文档的明确要求。
2. 文档内容与当前项目范围一致。
3. 文件保存在正确目录，命名清晰。
4. Git 状态清楚，无误提交 `deliverables/`。
5. 已交给审阅，确认后再提交并推送。

## 6. 项目风险管理

### 6.1 风险分级方法

使用概率 P 和影响 I 进行风险评估，取值 1 到 5：

`风险值 R = P * I`

| 风险值 | 等级 | 处理方式 |
|---:|---|---|
| 1-4 | 低 | 记录并定期观察 |
| 5-9 | 中 | 制定预防措施 |
| 10-16 | 高 | 重点跟踪并准备应急预案 |
| 17-25 | 很高 | 优先处理，必要时调整范围 |

### 6.2 主要风险和应对

| 编号 | 风险 | P | I | R | 等级 | 预防措施 | 应急预案 |
|---|---|---:|---:|---:|---|---|---|
| R1 | 需求范围扩大，超出课程周期 | 4 | 4 | 16 | 高 | 将需求分为核心、重要、扩展三类 | 放弃推荐匹配、通知等扩展，保留主流程 |
| R2 | 进度估计不足 | 4 | 4 | 16 | 高 | 每周更新跟踪表，任务拆到 1-2 天粒度 | 推迟低优先级优化，先保证可演示版本 |
| R3 | 文档与代码不同步 | 3 | 3 | 9 | 中 | 每个实验结束更新对应文档 | 交付前统一核对功能清单和截图/描述 |
| R4 | 文本数据文件损坏或格式不一致 | 3 | 4 | 12 | 高 | 保留样例数据，保存前校验字段 | 回退样例数据，必要时迁移 SQLite |
| R5 | 密码明文和个人信息安全不足 | 3 | 5 | 15 | 高 | 设计阶段明确哈希和最小化采集方案 | 演示数据全部使用模拟数据，后续补哈希 |
| R6 | 权限控制不足导致越权操作 | 3 | 5 | 15 | 高 | 角色入口和服务层都校验权限 | 发现越权时先关闭入口，再补服务层校验 |
| R7 | Qt 界面逻辑复杂，修改困难 | 3 | 3 | 9 | 中 | 界面层只处理交互，业务放入服务层 | 将重复逻辑提取为辅助函数 |
| R8 | 构建环境差异导致无法运行 | 2 | 4 | 8 | 中 | 记录 Qt/MinGW 路径，保留构建脚本 | 提供可运行构建产物或环境说明 |
| R9 | 测试不足导致演示失败 | 4 | 4 | 16 | 高 | 建立主流程回归测试清单 | 演示前按清单完整走一遍 |
| R10 | CASE 工具投入过重 | 3 | 3 | 9 | 中 | 优先使用 Markdown/Mermaid 轻量建模 | 减少重型工具建模，保留核心图表 |

### 6.3 风险管理结论

本项目风险总体可控。高风险主要集中在范围、进度、数据可靠性、安全和测试方面。只要坚持核心需求优先、短周期审阅、服务层封装和回归测试，风险不会从根本上否定项目可行性。

## 7. 可行性分析报告定稿

### 7.1 项目名称

人才招聘系统

### 7.2 项目背景

招聘活动涉及求职者、企业和平台管理者三类主体。传统手工或分散式管理方式存在信息更新慢、简历和岗位匹配效率低、审核流程不透明、留言反馈难以统一处理等问题。因此，开发一个轻量级人才招聘系统具有实际意义，也适合用于软件工程课程中的需求分析、建模、设计、实现、测试和项目管理训练。

### 7.3 功能范围

系统功能范围分为三类：

1. 个人用户功能：注册登录、个人资料维护、简历维护、岗位查询、岗位申请、撤销申请、留言。
2. 企业用户功能：注册登录、企业资料维护、岗位发布、岗位修改、岗位下架、人才查询、留言。
3. 管理员功能：审核个人用户、审核企业用户、回复留言、删除留言、修改密码。

扩展功能包括智能岗位推荐、通知提醒、数据库升级和统计图表，作为后续优化项。

### 7.4 技术路线

| 项目 | 方案 |
|---|---|
| 开发语言 | C++ |
| GUI 框架 | Qt Widgets |
| 业务层 | `RecruitmentService` |
| 数据存储 | 当前使用 `recruitment_data.txt`，后续可迁移 SQLite |
| 版本控制 | Git + GitHub |
| 文档管理 | Markdown + GitHub 仓库 |
| 建模工具 | StarUML、PlantUML/Mermaid |
| 进度工具 | GanttProject、GitHub Projects |

### 7.5 可行性结论

| 方面 | 结论 |
|---|---|
| 业务可行性 | 业务场景清晰，角色边界明确，主流程适合课程项目。 |
| 技术可行性 | Qt、C++、本地存储和服务层封装均已具备基础。 |
| 经济可行性 | 使用免费或已有工具，不需要额外服务器成本。 |
| 操作可行性 | 桌面图形界面易于演示和操作。 |
| 进度可行性 | 通过迭代开发和范围控制可以在课程周期内完成。 |
| 风险可控性 | 主要风险已识别，并有预防措施和应急预案。 |

综合判断：人才招聘系统具备较高可行性，适合作为软件工程课程项目继续推进。

## 8. 本次实验结论

通过实验三，项目过程从单纯“完成文档和代码”进一步明确为“适度前期设计 + Scrum 迭代 + Kanban 跟踪 + 局部 XP 实践”的混合过程。该方式既保留了传统软件工程对文档、阶段和风险的控制，又吸收了敏捷方法对反馈、迭代和可运行增量的重视。

可行性分析报告已经形成定稿，风险管理从简单列表细化为带概率、影响、等级、预防措施和应急预案的登记册。后续实验将在此基础上继续完成 SRS、建模、体系结构设计和模块设计评估。
