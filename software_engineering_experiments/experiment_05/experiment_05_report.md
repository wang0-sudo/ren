## 2. 活动图时间分析

### 2.1 计算方法

活动图分析采用关键路径法（CPM）：

1. 正向计算事件最早时间 E。
2. 反向计算事件最晚时间 L。
3. 对每条活动计算 ES、EF、LS、LF。
4. 活动时差 `Slack = LS - ES`。
5. 时差为 0 的活动构成关键路径。

## 3. 项目跟踪工具练习

### 3.1 工具方案

本项目采用轻量项目跟踪方案：

| 工具 | 用途 |
|---|---|
| GitHub Projects | 管理 Backlog、任务状态、优先级和审阅状态 |
| Mermaid Gantt | 在 Markdown 报告中表达甘特图 |
| Markdown 跟踪表 | 记录每周进度、风险和产出 |
| Git | 保存每次实验成果和提交历史 |

### 3.2 甘特图和任务状态

项目甘特图、任务表和周更新记录见 `project_tracking_gantt.md`。

本阶段建议跟踪以下核心指标：

1. 每个实验是否有独立成果目录。
2. 每个成果是否经过审阅后再推送。
3. 当前是否存在阻塞任务。
4. 关键路径任务是否延期。
5. 风险登记册是否每周更新。

### 3.3 跟踪结论

对于课程项目，项目跟踪工具不需要过度复杂。最重要的是让任务、状态、负责人角色、风险和产出文件可见。GitHub Projects 适合做看板，Markdown 适合做报告留档，Git 适合做历史追踪。

## 4. 软件开发团队组织结构对比

### 4.1 常见组织结构

| 结构 | 特点 | 优点 | 局限 |
|---|---|---|---|
| 职能型 | 按需求、开发、测试、运维等专业划分 | 专业能力集中 | 交接成本高 |
| 项目型 | 围绕项目建立团队 | 目标集中，响应快 | 专业资源可能重复 |
| 矩阵型 | 同时有职能线和项目线 | 兼顾专业沉淀和项目交付 | 容易多头管理 |
| 敏捷跨职能团队 | 产品、开发、测试等共同对迭代负责 | 反馈快，责任清晰 | 要求较强自组织能力 |

### 4.2 国内外团队工作方式对比

| 维度 | 国内常见方式 | 国外常见方式 | 分析 |
|---|---|---|---|
| 管理风格 | 更强调执行速度和结果 | 更强调节奏、流程和边界 | 应结合项目目标和可持续性 |
| 沟通方式 | 即时沟通较多 | 异步文档和会议边界较清晰 | 关键结论应沉淀为文档 |
| 工程实践 | 团队差异较大 | CI/CD、代码评审、测试自动化更普遍 | 本项目应建立轻量工程规范 |
| 工作时间 | 部分团队存在长时间加班 | 更重视工作生活边界 | 长期高强度工作不利于质量 |
| 团队角色 | 项目经理或技术负责人推动较多 | Product Owner、Scrum Master、Developers 划分更明确 | 课程项目可一人多角色，但职责要清楚 |

### 4.3 对 996 工作制的看法

从软件工程角度看，996 不应作为长期稳定的管理方式。软件开发是知识密集型工作，长期高强度加班会带来以下问题：

1. 缺陷率上升，返工增加。
2. 沟通质量下降，需求误解增多。
3. 技术债堆积，系统维护困难。
4. 团队流失风险上升。
5. 项目管理问题被加班掩盖。

更合理的方式是通过明确优先级、缩小范围、自动化测试、持续集成和复盘来提升效率。

## 5. 个人偏好的工作方式

个人更偏好目标清晰、节奏可持续、沟通有记录、工具链明确的工作方式。

理想环境包括：

1. 任务有明确验收标准。
2. 有稳定的版本控制和构建方式。
3. 重要结论写入文档。
4. 允许对明显不合理的设计进行重构。
5. 加班是短期例外，而不是长期制度。

可以接受的约束包括阶段性截止日期、代码规范、文档模板、审阅流程和回归测试要求。

## 6. 项目组管理建议

对人才招聘系统项目，建议采用轻量敏捷项目制：

1. 使用项目负责人维护计划、风险和提交节奏。
2. 使用需求负责人维护 SRS 和用例。
3. 使用设计负责人维护数据模型、活动图和架构。
4. 使用开发负责人维护 Qt 界面和服务层。
5. 使用测试负责人维护回归测试清单。
6. 使用文档负责人整理实验报告。

如果实际人数较少，可以一人兼任多个角色，但仍应保持职责清晰。

## 7. 本次实验结论

通过实验五，活动图从“图形理解”进一步落实为可计算的进度分析工具。实验五活动图的关键路径为 `A -> D -> H -> L -> N -> O -> P`，总长度为 47。项目跟踪方面，GitHub Projects、Markdown 跟踪表、Mermaid Gantt 和 Git 可以组成低成本、可追溯的管理方案。团队组织方面，本项目更适合轻量敏捷跨职能团队，以清晰角色、持续审阅和回归测试保障交付。
