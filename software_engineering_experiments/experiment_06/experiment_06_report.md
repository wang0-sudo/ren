## 2. 工作量估算

### 2.1 估算依据

当前人才招聘系统核心代码主要包括：

1. `RecruitmentService` 服务层。
2. `LoginWindow` 登录注册界面。
3. `DashboardWindow` 三类角色主界面。
4. `main.cpp` 程序入口。

核心代码约 2470 行，估算规模取 `2.5 KSLOC`。

### 2.2 COCOMO II 简化估算

采用公式：

`PM = A * Size^E * EAF`

本项目取值：

| 参数 | 取值 |
|---|---:|
| A | 2.94 |
| Size | 2.5 KSLOC |
| E | 1.08 |
| EAF | 0.90 |

计算：

`PM = 2.94 * 2.5^1.08 * 0.90 ≈ 7.1 人月`

因此，从正式开发模型角度估算，人才招聘系统完整开发约为 7.1 人月。

### 2.3 课程项目修正

考虑已有原型、课程交付范围和轻量化要求，课程剩余工作量约为 4.2 人月。若 3 人小组每人每周投入约 0.25 人月，则预计约 5.6 周可完成核心交付。

详细估算见 `effort_estimation.md`。

## 3. 风险管理细化

### 3.1 风险分级

风险值：

`R = P * I`

| 风险值 | 等级 | 处理方式 |
|---:|---|---|
| 1-4 | 低 | 记录并复查 |
| 5-9 | 中 | 制定预防措施 |
| 10-16 | 高 | 每周跟踪并准备应急预案 |
| 17-25 | 很高 | 优先处理并调整范围 |

### 3.2 高风险项

| 编号 | 风险 | R | 应对重点 |
|---|---|---:|---|
| R1 | 功能范围持续扩大 | 16 | 核心/重要/扩展三级分类 |
| R2 | 文档和代码任务堆积 | 16 | 每个实验完成后立即审阅和推送 |
| R4 | 密码明文保存 | 15 | 使用模拟数据，后续补密码哈希 |
| R5 | 角色越权 | 15 | 界面入口和服务层双重校验 |
| R6 | 主流程缺少回归测试 | 16 | 建立登录、注册、审核、岗位、留言测试清单 |
| R10 | SRS/SAD 与实际功能不一致 | 12 | 文档章节映射实际功能 |

完整风险计划见 `risk_management_plan.md`。

## 4. SRS 文档结构学习

### 4.1 SRS 的作用

SRS 是软件需求规格说明，用于描述系统必须满足的功能需求、非功能需求、接口需求、数据需求和验收标准。它是设计、实现、测试和验收的共同依据。

### 4.2 SRS 与相关文档关系

| 文档 | 关系 |
|---|---|
| 可行性分析报告 | 判断项目是否值得做 |
| SRS | 明确系统要做什么 |
| SDD/SAD | 根据 SRS 设计系统怎么做 |
| 测试文档 | 根据 SRS 验证系统是否满足需求 |
| 用户手册 | 将 SRS 中用户功能转化为操作说明 |

### 4.3 人才招聘系统 SRS 重点章节

| 章节 | 重点 |
|---|---|
| 引言 | 编写目的、背景、术语、参考资料 |
| 总体描述 | 产品功能、用户特征、运行环境、约束 |
| 功能需求 | 个人用户、企业用户、管理员需求 |
| 数据需求 | 用户、简历、企业、岗位、留言 |
| 外部接口需求 | Qt 界面、本地文件、操作系统环境 |
| 非功能需求 | 易用性、可靠性、安全性、可维护性 |
| 验收标准 | 主流程可运行、数据可保存、权限正确 |

详细结构见 `srs_structure.md`。
