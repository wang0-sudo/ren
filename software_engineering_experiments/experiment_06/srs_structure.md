# 人才招聘系统 SRS 文档结构学习

## 1. SRS 的作用

SRS（Software Requirements Specification，软件需求规格说明）用于准确描述软件系统应满足的功能、性能、接口、约束和质量属性。它连接可行性分析、设计说明、测试计划和最终验收，是后续设计和实现的重要依据。

## 2. SRS 与其他文档关系

| 文档 | 与 SRS 的关系 |
|---|---|
| 可行性分析报告 | 决定项目是否值得做，SRS 进一步明确要做什么。 |
| 项目开发计划 | 根据 SRS 的范围安排进度、人员和里程碑。 |
| 软件设计说明 SDD/SAD | 根据 SRS 的需求设计架构、模块和接口。 |
| 测试计划/测试用例 | 根据 SRS 编写验收标准和测试场景。 |
| 用户手册 | 将 SRS 中的用户功能转化为操作说明。 |

### 4 数据需求

| 数据对象 | 主要字段 |
|---|---|
| PersonalUser | id、username、password、name、gender、phone、email、resume、appliedPositionIds、status |
| Resume | education、experience、skills、selfIntroduction |
| EnterpriseUser | id、username、password、companyName、contactPerson、phone、email、address、introduction、status |
| Position | id、enterpriseId、title、location、salary、requirement、description、active |
| Message | id、senderType、senderName、content、handled、reply |

### 5 外部接口需求

| 接口 | 说明 |
|---|---|
| 用户界面 | Qt Widgets 桌面窗口，包括登录窗口和三类角色工作台。 |
| 文件接口 | 使用 `recruitment_data.txt` 保存和读取数据。 |
| 操作系统接口 | Windows 桌面环境、Qt/MinGW 运行库。 |

### 6 非功能需求

| 类型 | 需求 |
|---|---|
| 易用性 | 界面应提供清晰按钮、表格和错误提示。 |
| 可靠性 | 保存失败应给出提示，重启后数据可恢复。 |
| 安全性 | 不同角色只能访问对应功能；后续应支持密码哈希。 |
| 可维护性 | 业务规则集中在服务层，界面层负责交互。 |
| 可移植性 | 在 Windows + Qt 环境下可构建运行。 |
| 性能 | 课程规模数据下查询和刷新应基本即时完成。 |

### 7 验收标准

| 编号 | 验收项 |
|---|---|
| AC-01 | 个人和企业均可注册，注册后进入待审核状态。 |
| AC-02 | 管理员可以审核个人和企业用户。 |
| AC-03 | 个人用户可以维护简历并申请岗位。 |
| AC-04 | 企业用户可以发布、修改和下架岗位。 |
| AC-05 | 管理员可以处理留言。 |
| AC-06 | 重启程序后核心数据仍可读取。 |
| AC-07 | 文档、模型和代码描述保持一致。 |

## 5. 学习结论

SRS 的核心不是堆砌模板，而是用可验证的方式描述“系统必须做什么”。对人才招聘系统来说，SRS 应重点明确三类角色、核心业务流程、数据对象、权限边界、非功能需求和验收标准。后续实验七可以直接以本结构为基础扩展为 SRS 初稿。
