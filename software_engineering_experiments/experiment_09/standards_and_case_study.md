# 标准对比与经典体系结构案例研究

## 1. GB 软件设计说明与 ISO/IEC/IEEE 42010 对比

### 1.1 GB 软件设计说明侧重点

GB 风格的软件设计说明通常强调实现落地，关注：

1. 引言和总体设计。
2. 软件结构和模块划分。
3. 接口设计。
4. 数据结构设计。
5. 运行设计。
6. 出错处理设计。
7. 安全保密设计。
8. 维护设计。

该结构适合课程项目和工程交付，因为它能直接指导开发人员进行模块实现、接口调用和数据处理。

### 1.2 ISO/IEC/IEEE 42010 侧重点

ISO/IEC/IEEE 42010:2022 是软件、系统和企业架构描述标准。IEEE 和 ISO 页面均说明，该标准定义架构描述的结构和表达要求，强调架构描述应围绕利益相关者、关注点、视点、视图、模型、对应关系和架构理由展开，并且不规定具体记录格式。

对本项目有价值的思想包括：

1. 区分“架构本身”和“架构描述”。
2. 先识别利益相关者和关注点，再选择视点和视图。
3. 每个视图服务于明确的关注点。
4. 需要记录不同视图之间的对应关系和一致性规则。
5. 需要说明关键架构决策理由。

参考：

1. IEEE SA: https://standards.ieee.org/ieee/42010/6846/
2. ISO: https://www.iso.org/standard/74393.html

### 1.3 对比表

| 对比项 | GB 软件设计说明 | ISO/IEC/IEEE 42010 |
|---|---|---|
| 核心目标 | 指导软件设计和实现 | 规范架构描述的表达方式 |
| 文档组织 | 偏固定章节 | 不固定格式，强调概念要素 |
| 关注对象 | 模块、接口、数据、运行和错误处理 | 利益相关者、关注点、视点、视图、模型和对应关系 |
| 适合场景 | 课程设计、工程详细设计、实现交付 | 架构沟通、评审、长期演进和跨团队协作 |
| 对本项目用途 | 作为 SAD 章节骨架 | 作为多视图和决策记录依据 |

### 1.4 本项目采用方式

人才招聘系统 SAD 采用 GB 风格章节保证可读性和可执行性，同时采用 ISO/IEC/IEEE 42010 的视点/视图方法组织架构模型。最终形成的文档包括：

1. 架构目标和约束。
2. 利益相关者与关注点。
3. 上下文视图、逻辑视图、组件视图、运行视图、数据视图和部署视图。
4. 组件职责和接口设计。
5. 架构决策、风险和一致性规则。

## 2. 经典体系结构案例研究

### 2.1 Parnas 模块分解思想

经典模块化思想强调模块划分不应只按流程步骤划分，而应按可能变化的设计决策进行信息隐藏。对人才招聘系统的启示如下：

| 变化点 | 应隐藏的位置 | 当前设计 |
|---|---|---|
| 数据文件格式 | `RecruitmentService::load()`、`save()` | 界面层不解析文件。 |
| 岗位可见规则 | `RecruitmentService::isPositionVisible()` | 个人岗位列表和申请逻辑共享同一规则。 |
| 用户名唯一规则 | `RecruitmentService::usernameExists()` | 注册逻辑统一调用。 |
| 审核状态规则 | `RecruitmentService` 审核和申请/发布方法 | UI 只展示结果，不复制业务规则。 |

后续如果从文本文件迁移到 SQLite，理想情况下主要修改业务层的数据访问实现，而不需要大规模修改窗口代码。

### 2.2 分层架构案例

分层架构将系统划分为表示层、业务层和数据层。它适合本项目的原因：

1. 用户交互由 Qt 窗口类处理。
2. 业务规则集中在 `RecruitmentService`。
3. 数据保存集中在本地文件读写方法。

优点：

1. 结构清晰，便于课程展示。
2. 业务规则不散落在多个按钮事件中。
3. 后续测试可直接围绕服务层方法设计。

局限：

1. 当前业务服务类较集中，继续扩展后会变大。
2. 数据层还不是独立接口，替换存储时仍需修改 `RecruitmentService`。

### 2.3 4+1 视图模型

4+1 视图模型用逻辑视图、开发视图、进程视图、物理视图和场景视图共同描述架构。对本项目的映射如下：

| 4+1 视图 | 本项目对应内容 |
|---|---|
| 逻辑视图 | `PersonalUser`、`EnterpriseUser`、`Position`、`Message`、`RecruitmentService` |
| 开发视图 | `qt_gui` 源文件、CMake 配置、构建脚本 |
| 进程视图 | 单进程 Qt 事件循环、窗口与服务调用序列 |
| 物理视图 | Windows 桌面环境、Qt 运行库、本地数据文件 |
| 场景视图 | 注册登录、审核、发布岗位、申请岗位、留言处理 |

本项目规模较小，因此没有单独引入多进程或服务端部署视图，但仍可用多视图方式提高架构表达清晰度。

### 2.4 MVC/MVP 思想

Qt Widgets 项目中常见的 GUI 组织方式与 MVC/MVP 思想相近：

| 角色 | 本项目对应 |
|---|---|
| View | `LoginWindow`、`DashboardWindow` 中的控件和表格 |
| Presenter/Controller | 窗口类中的事件处理函数 |
| Model/Service | `RecruitmentService` 和领域结构体 |

当前窗口类同时承担 View 和 Controller 职责。对于课程规模系统可以接受；若继续扩展，可进一步拆出控制器或 ViewModel，减少窗口类体积。

### 2.5 依赖注入思想

当前 `LoginWindow` 和 `DashboardWindow` 都通过构造函数接收 `std::shared_ptr<RecruitmentService>`。这体现了依赖注入的基本思想：

1. 窗口类不直接创建服务对象。
2. 多个窗口共享同一个服务实例和数据状态。
3. 后续测试或重构时可以替换服务实现。

改进方向：

1. 抽象出服务接口，例如 `IRecruitmentService`。
2. 将数据存储能力拆成 `RecruitmentRepository`。
3. 在测试中注入内存版仓储，减少对真实文件的依赖。

## 3. 对本项目 SAD 的具体影响

| 来源 | 影响 |
|---|---|
| GB 软件设计说明 | SAD 保留模块设计、接口设计、数据设计、运行设计和出错处理。 |
| ISO/IEC/IEEE 42010 | SAD 增加利益相关者、关注点、视点、视图、对应关系和架构决策。 |
| Parnas 模块分解 | 将文件格式、岗位可见性、用户名唯一性等变化点封装在服务层。 |
| 分层架构 | 明确表示层、业务层、数据层职责。 |
| 4+1 视图 | 使用多个视图表达同一架构，避免单一图无法覆盖所有关注点。 |
| 依赖注入 | 保留构造函数注入服务实例的设计，并作为后续测试改进基础。 |
