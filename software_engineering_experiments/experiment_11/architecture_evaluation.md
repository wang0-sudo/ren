# 人才招聘系统架构评估

## 1. 评估目标

本评估用于判断当前体系结构是否满足课程项目需求，并识别后续改进方向。评估重点包括：

1. 架构是否支持 SRS 中的核心业务流程。
2. 组件职责是否清晰。
3. 关键质量属性是否达到课程项目要求。
4. 当前风险是否被识别并有可行应对方案。

## 2. 评估方法

采用轻量级质量属性场景分析方法：

1. 选取主要质量属性。
2. 为每个质量属性定义场景。
3. 分析当前架构响应方式。
4. 记录风险、权衡和改进建议。

## 3. 质量属性场景

### 3.1 可维护性

| 项目 | 内容 |
|---|---|
| 场景 | 后续需要修改岗位可见规则，例如增加“岗位审核状态”。 |
| 刺激 | 开发人员修改需求。 |
| 环境 | 系统正常维护阶段。 |
| 响应 | 修改 `RecruitmentService::isPositionVisible()`，并更新相关测试和文档。 |
| 评价 | 当前岗位可见规则集中在服务层，修改范围较可控。 |
| 风险 | 如果窗口层复制岗位可见规则，会产生不一致。 |

### 3.2 可靠性

| 项目 | 内容 |
|---|---|
| 场景 | 用户提交岗位申请后关闭程序，再次启动仍能看到已申请岗位。 |
| 刺激 | 用户执行申请操作并重启系统。 |
| 环境 | 本地文件系统可写。 |
| 响应 | `applyForPosition()` 更新数据后调用 `persist()`，重启时 `load()` 读取文件。 |
| 评价 | 当前设计满足课程规模的基本可靠性。 |
| 风险 | 文本文件写入过程中异常中断可能造成文件损坏。 |

### 3.3 安全性

| 项目 | 内容 |
|---|---|
| 场景 | 未审核企业用户尝试发布岗位。 |
| 刺激 | 企业用户点击发布岗位。 |
| 环境 | 企业用户状态为 `Pending` 或 `Rejected`。 |
| 响应 | `addPosition()` 检查企业审核状态，不通过则返回错误。 |
| 评价 | 关键业务准入在服务层校验，不只依赖 UI。 |
| 风险 | 密码明文保存、本地文件未加密。 |

### 3.4 可测试性

| 项目 | 内容 |
|---|---|
| 场景 | 测试同一个个人用户不能重复申请同一岗位。 |
| 刺激 | 测试代码连续调用两次 `applyForPosition()`。 |
| 环境 | 个人用户审核通过，岗位可见。 |
| 响应 | 第一次返回成功，第二次返回失败并给出错误信息。 |
| 评价 | 规则集中在 `RecruitmentService`，适合服务层测试。 |
| 风险 | 当前未建立完整自动化测试工程。 |

### 3.5 可扩展性

| 项目 | 内容 |
|---|---|
| 场景 | 后续将文本文件存储替换为 SQLite。 |
| 刺激 | 数据量增大或需要更稳定的数据管理。 |
| 环境 | 系统进入改进阶段。 |
| 响应 | 抽取仓储接口，将 `load()`、`save()` 中的文件读写迁移到 SQLite 实现。 |
| 评价 | 当前 UI 不直接依赖文件格式，有利于迁移。 |
| 风险 | `RecruitmentService` 同时承担业务和存储职责，迁移前需要拆分。 |

### 3.6 易用性

| 项目 | 内容 |
|---|---|
| 场景 | 管理员需要快速处理待审核个人用户和企业用户。 |
| 刺激 | 管理员登录工作台。 |
| 环境 | 存在待审核用户。 |
| 响应 | `DashboardWindow` 按角色显示管理员标签页，列表中展示待审核对象。 |
| 评价 | 角色化工作台满足课程演示使用。 |
| 风险 | 数据量增大后表格筛选和分页能力不足。 |

## 4. 架构风险清单

| 编号 | 风险 | 严重度 | 可能性 | 说明 | 应对 |
|---|---|---|---|---|---|
| R-01 | 文本文件存储缺少事务 | 中 | 中 | 异常中断时可能影响数据完整性 | 后续迁移 SQLite 或增加备份写入。 |
| R-02 | 密码明文存储 | 中 | 高 | 不符合真实系统安全要求 | 增加哈希和盐值。 |
| R-03 | `RecruitmentService` 职责集中 | 中 | 中 | 功能继续增加后维护成本上升 | 拆分服务和仓储。 |
| R-04 | 自动化测试不足 | 中 | 高 | 回归依赖人工验证 | 增加服务层单元测试。 |
| R-05 | 构建路径绑定本机 Qt 路径 | 低 | 中 | 换机器可能需要修改脚本 | 使用环境变量或 CMake Presets。 |

## 5. 权衡分析

### 5.1 简单实现与可扩展性

当前使用文本文件存储，优点是实现简单、便于演示，缺点是事务能力弱、查询能力弱。对于课程项目是可接受的；若项目继续扩大，应迁移 SQLite。

### 5.2 集中式服务与职责拆分

将业务规则集中在 `RecruitmentService` 可以降低 UI 复杂度，也方便测试。但随着功能增长，该类可能变大。当前阶段保持集中式服务可以减少过早拆分；后续可逐步拆分为用户服务、岗位服务、留言服务和仓储。

### 5.3 桌面端与 Web 化

Qt 桌面端适合当前环境和课程演示，部署简单、交互直观。但不适合多人在线协作。若后续要变成真实招聘平台，应新增服务端和数据库，而不是继续扩展单机文件方案。

## 6. 评估结论

当前架构满足课程项目的主要质量目标：

1. 功能闭环完整。
2. 组件职责基本清晰。
3. 业务规则集中于服务层。
4. 三类用户工作台与业务流程匹配。
5. 文档模型和代码结构能够互相追踪。

当前架构不适合作为真实互联网招聘平台直接使用，主要原因是单机桌面部署、文本文件存储、明文密码和缺少并发控制。但这些限制与课程项目边界一致，并已在 SAD 中记录为后续改进方向。
