# 架构改进计划

## 1. 改进目标

在保持当前课程项目可运行和可提交的基础上，规划后续架构演进路线。改进目标包括：

1. 降低 `RecruitmentService` 职责集中带来的维护成本。
2. 提高数据存储可靠性和安全性。
3. 增强测试能力。
4. 为后续功能扩展保留清晰路径。

## 2. 短期改进

短期目标是支撑课程提交和答辩。

| 改进项 | 优先级 | 工作内容 | 预期收益 |
|---|---|---|---|
| 完成 SAD 定稿 | 高 | 保持 SRS、SAD、模型和代码一致 | 满足实验十一提交要求。 |
| 建立评审清单 | 高 | 用 `sad_review_checklist.md` 检查文档完整性 | 降低遗漏风险。 |
| 补充测试点 | 高 | 基于 SRS 和 SAD 整理注册、审核、岗位、申请、留言测试点 | 支撑后续验证。 |
| 明确限制说明 | 中 | 记录文本存储和明文密码风险 | 答辩时边界清晰。 |

## 3. 中期改进

中期目标是提升代码可维护性和可测试性。

### 3.1 抽取仓储层

当前 `RecruitmentService` 同时负责业务规则和文件读写。建议新增：

```text
RecruitmentRepository
FileRecruitmentRepository
```

职责：

1. `RecruitmentService` 保留业务规则。
2. `RecruitmentRepository` 负责数据读取和保存接口。
3. `FileRecruitmentRepository` 实现当前文本文件存储。

预期收益：

1. 服务层更容易测试。
2. 数据存储替换成本降低。
3. 文件格式变化不影响业务方法。

### 3.2 增加服务层测试

优先测试以下规则：

1. 重复用户名注册失败。
2. 未审核个人用户不能申请岗位。
3. 未审核企业用户不能发布岗位。
4. 下架岗位不可见。
5. 重复申请同一岗位失败。
6. 留言和回复不能为空。
7. 保存后可重新加载。

### 3.3 拆分业务服务

当功能继续扩展时，可拆分为：

| 服务 | 职责 |
|---|---|
| `UserService` | 注册、登录、审核、密码修改。 |
| `PositionService` | 岗位发布、修改、下架、可见性查询。 |
| `ApplicationService` | 岗位申请和撤销。 |
| `MessageService` | 留言提交、回复和删除。 |
| `PersistenceService` | 数据加载和保存。 |

## 4. 长期改进

长期目标是从课程演示系统演进为更接近真实招聘平台的系统。

| 改进项 | 说明 |
|---|---|
| 迁移 SQLite | 提升数据一致性、查询能力和维护性。 |
| 密码哈希 | 使用哈希和盐值替代明文密码。 |
| 操作日志 | 记录管理员审核、岗位管理和留言处理操作。 |
| 配置化路径 | 将数据文件路径、Qt 环境路径等从代码和脚本中配置化。 |
| 服务端化 | 若需要多人在线使用，可新增后端 API 和数据库。 |
| 权限模型细化 | 将角色权限抽象为权限表或策略对象。 |

## 5. 改进优先级矩阵

| 改进项 | 价值 | 成本 | 建议 |
|---|---|---|---|
| 服务层测试 | 高 | 中 | 优先实施。 |
| 抽取仓储层 | 高 | 中 | 测试之后实施。 |
| SQLite 迁移 | 高 | 中 | 中期实施。 |
| 密码哈希 | 中 | 低 | 中期实施。 |
| 拆分业务服务 | 中 | 中 | 功能增长后实施。 |
| 服务端化 | 高 | 高 | 仅在项目目标变化时实施。 |

## 6. 演进路线图

```mermaid
flowchart LR
    A[当前三层架构] --> B[补充服务层测试]
    B --> C[抽取仓储接口]
    C --> D[迁移 SQLite]
```

## 7. 结论

当前架构适合课程阶段，不需要为了短期提交进行大规模重构。推荐策略是先完成 SAD 和测试材料，再在后续迭代中逐步抽取仓储层、补充服务层测试和改进安全存储。
