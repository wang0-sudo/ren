4. 更新项目跟踪表，记录体系结构设计阶段的完成情况。

## 3. SAD 定稿工作

实验九已形成 SAD 初稿，本次实验进一步补充和定稿：

1. 明确文档范围和提交对象。
2. 补充架构背景、设计原则和约束。
3. 完善利益相关者、关注点、架构视图和视图一致性。
4. 细化组件职责、接口设计、数据设计和运行流程。
5. 增加错误处理、安全设计、维护设计和可测试性设计。
6. 形成架构决策记录和后续演进建议。

## 4. 架构评估方法

本次评估采用轻量级质量属性场景分析方法，围绕以下质量属性进行：

| 质量属性 | 关注问题 |
|---|---|
| 可维护性 | 需求变化时修改范围是否可控。 |
| 可靠性 | 数据保存、加载和错误处理是否稳定。 |
| 安全性 | 角色隔离、审核准入和密码存储是否满足课程项目要求。 |
| 可测试性 | 核心业务规则是否能被设计为明确测试用例。 |
| 可扩展性 | 后续扩展数据库、服务拆分或新功能是否有清晰路径。 |
| 易用性 | 三类用户工作台是否符合业务使用路径。 |

评估结果表明：当前三层架构适合课程演示和现有功能规模，主要风险集中在文本文件存储、明文密码、服务类职责集中和自动化测试不足。

## 5. 改进结论

短期改进应优先保障提交质量，包括：

1. 保持 SRS、SAD、代码和模型一致。
2. 用评审清单检查 SAD 完整性。
3. 在后续总结中明确当前架构的边界和风险。

中期改进可围绕测试和存储展开：

1. 为 `RecruitmentService` 增加单元测试。
2. 将数据访问拆为仓储类。
3. 将文本文件替换为 SQLite。
4. 增加密码哈希存储。

## 6. 实验结论

本次实验完成了人才招聘系统 SAD 定稿，并对架构进行了质量属性评估。当前架构采用 Qt Widgets 单机桌面端、表示层/业务层/数据层三层结构、共享 `RecruitmentService` 服务实例和本地文本文件存储。该设计能够满足课程项目演示、文档提交和后续答辩需求，同时也清楚记录了后续可改进方向。
