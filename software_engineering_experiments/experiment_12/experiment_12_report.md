# 实验十二报告：设计原则与依赖注入

## 1. 实验目的

本次实验围绕人才招聘系统的模块设计进行评估，学习依赖注入技术，并结合项目实践分析面向对象设计原则。实验目标如下：

1. 培养对设计原则进行实践评估的能力。
2. 学习依赖注入的基本思想、价值和使用场景。
3. 理解并应用利斯科夫替换原则、单一职责原则、开闭原则、迪米特法则、依赖倒转原则和合成复用原则。
4. 结合当前项目模块发现设计问题并提出改进方案。
5. 更新项目跟踪表，记录设计模块阶段工作。

## 2. 项目模块概况

当前人才招聘系统采用 Qt Widgets 桌面端架构，核心模块如下：

| 模块 | 主要职责 |
|---|---|
| `main.cpp` | 初始化 Qt 应用、创建共享业务服务、显示登录窗口。 |
| `LoginWindow` | 处理角色选择、登录、个人注册和企业注册。 |
| `DashboardWindow` | 按角色展示工作台，处理个人、企业和管理员操作。 |
| `RecruitmentService` | 管理用户、岗位、留言和持久化，执行业务规则。 |
| 领域结构体 | `PersonalUser`、`EnterpriseUser`、`Position`、`Message`、`Resume`。 |
| `recruitment_data.txt` | 保存系统业务数据。 |

## 3. 设计原则学习与应用

本次实验对六项原则进行了项目化分析：

| 原则 | 当前项目体现 | 主要问题 | 改进方向 |
|---|---|---|---|
| 单一职责原则 | UI 和业务服务已有初步分离 | `DashboardWindow` 和 `RecruitmentService` 职责偏多 | 拆分角色面板和业务服务。 |
| 开闭原则 | 新增业务可通过服务方法扩展 | 当前新增字段需修改保存/加载多处代码 | 抽取仓储层和序列化逻辑。 |
| 里氏替换原则 | 当前继承主要来自 Qt 窗口体系 | 业务层缺少接口抽象，替换空间有限 | 引入服务接口或仓储接口。 |
| 迪米特法则 | 窗口主要通过服务接口访问数据 | 窗口仍直接接触领域结构体细节 | 引入视图模型或 DTO。 |
| 依赖倒转原则 | 窗口通过构造函数接收服务对象 | 依赖仍指向具体 `RecruitmentService` | 抽象为 `IRecruitmentService`。 |
| 合成复用原则 | 窗口组合 Qt 控件和服务对象 | 没有滥用继承扩展业务 | 继续优先使用组合。 |

详细分析见 `design_principles_analysis.md`。

## 4. 依赖注入学习总结

依赖注入的核心思想是：对象不在内部直接创建依赖，而是由外部将依赖传入。这样可以降低对象与具体依赖的耦合，提高测试和替换能力。

本项目已有依赖注入的雏形：

```cpp
LoginWindow(std::shared_ptr<RecruitmentService> service, QWidget* parent = nullptr);
DashboardWindow(std::shared_ptr<RecruitmentService> service,
                UserRole role,
                int currentUserId,
                QWidget* parent = nullptr);
```

该设计的优点：

1. `LoginWindow` 和 `DashboardWindow` 不负责创建 `RecruitmentService`。
2. 多个窗口可以共享同一个服务实例。
3. 服务生命周期由 `main.cpp` 统一管理。
4. 后续可将服务替换为测试替身或接口实现。

当前不足是依赖类型仍为具体类。后续可引入 `IRecruitmentService` 或仓储接口，使窗口依赖抽象而非具体实现。

## 5. 模块设计评估结论

当前模块设计能够满足课程项目功能闭环和演示要求，主要优点包括：

1. 表示层与业务层已有明确边界。
2. 业务规则集中在 `RecruitmentService`。
3. 窗口类通过构造函数注入服务对象。
4. 数据文件读写未暴露给界面层。
5. 核心对象与 SRS、SAD 保持一致。

主要问题包括：

1. `DashboardWindow` 同时处理三类角色界面，类体积较大。
2. `RecruitmentService` 同时承担业务规则、数据集合管理和文件持久化。
3. 保存/加载逻辑与领域对象强绑定，字段扩展需要同步修改多处。
4. 缺少服务接口和仓储接口，自动化测试替换能力有限。

## 6. 改进方案

短期不建议为课程提交进行大规模重构，应保持系统稳定。推荐后续按以下顺序改进：

1. 为 `RecruitmentService` 增加服务层测试。
2. 抽取 `RecruitmentRepository`，分离业务规则和存储实现。
3. 拆分 `DashboardWindow` 为个人、企业、管理员三个子面板。
4. 引入 `IRecruitmentService`，让窗口依赖抽象接口。
5. 将文本文件存储迁移为 SQLite，并增加密码哈希。

## 7. 实验结论

通过本次实验，人才招聘系统的模块设计得到了从设计原则角度的系统评估。当前设计已经体现了分层、组合和构造函数注入思想，能够满足课程项目要求；同时也暴露出职责集中、抽象不足和测试替换能力有限等问题。后续可在保持系统稳定的前提下，逐步通过接口抽象、仓储层、服务拆分和测试补强提升设计质量。
