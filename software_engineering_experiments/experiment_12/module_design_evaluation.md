# 模块设计评估

## 1. 评估范围

本次评估覆盖人才招聘系统当前核心模块：

1. `main.cpp`
2. `LoginWindow`
3. `DashboardWindow`
4. `RecruitmentService`
5. 领域结构体
6. 本地数据文件与构建脚本

## 2. `main.cpp`

### 2.1 当前职责

1. 初始化 Qt 应用。
2. 创建 `RecruitmentService`。
3. 加载或生成数据。
4. 创建登录窗口。

### 2.2 设计评价

优点：

1. 启动流程简单清晰。
2. 服务对象创建位置集中。
3. 具备组合根雏形。

问题：

1. 如果后续依赖增多，`main.cpp` 会承担更多对象组装责任。

建议：

1. 当前保持不变。
2. 后续可抽取 `ApplicationBootstrapper` 或 `CompositionRoot`。

## 3. `LoginWindow`

### 3.1 当前职责

1. 显示角色、用户名和密码输入。
2. 处理登录。
3. 处理个人注册和企业注册。
4. 登录成功后创建工作台窗口。

### 3.2 设计评价

优点：

1. 职责相对集中。
2. 通过构造函数接收服务对象。
3. 登录和注册入口明确。

问题：

1. 注册表单构建和校验逻辑仍在窗口类中。
2. 登录成功后直接创建 `DashboardWindow`，窗口间跳转逻辑和登录逻辑耦合。

建议：

1. 若注册表单继续复杂化，可抽取 `PersonalRegisterDialog` 和 `EnterpriseRegisterDialog`。
2. 若窗口流程更多，可引入简单导航控制器。

## 4. `DashboardWindow`

### 4.1 当前职责

1. 按角色构建不同标签页。
2. 展示个人、企业、管理员数据。
3. 响应个人用户操作。
4. 响应企业用户操作。
5. 响应管理员操作。
6. 刷新表格和提示信息。

### 4.2 设计评价

优点：

1. 统一工作台便于课程演示。
2. 角色功能集中在一个窗口，操作路径直观。
3. 业务操作通过服务层完成。

问题：

1. 类体积较大，修改任一角色界面都可能影响同一文件。
2. 个人、企业、管理员事件处理函数集中在一个类中。
3. 表格填充逻辑和业务操作触发逻辑混在一起。

建议：

1. 拆分为 `PersonalDashboardPanel`、`EnterpriseDashboardPanel`、`AdminDashboardPanel`。
2. 将表格填充提取为辅助方法或视图模型。
3. 保留 `DashboardWindow` 作为容器窗口。

## 5. `RecruitmentService`

### 5.1 当前职责

1. 管理个人用户、企业用户、岗位和留言集合。
2. 提供注册、登录、审核、岗位、申请、留言和密码修改方法。
3. 执行业务规则校验。
4. 保存和加载本地数据文件。
5. 生成样例数据。

### 5.2 设计评价

优点：

1. 业务规则集中，UI 不直接处理核心规则。
2. 提供清晰的公开接口。
3. 数据文件格式封装在服务内部。
4. 适合围绕服务层设计测试。

问题：

1. 同时负责业务和存储，违反严格单一职责。
2. 保存/加载代码较长，字段变化影响较大。
3. 缺少接口抽象，不便于替换实现。

建议：

1. 抽取 `RecruitmentRepository`。
2. 将样例数据生成拆成 `DemoDataFactory`。
3. 后续按业务域拆分服务。

## 6. 领域结构体

### 6.1 当前职责

领域结构体主要承载数据：

1. `PersonalUser`
2. `Resume`
3. `EnterpriseUser`
4. `Position`
5. `Message`

### 6.2 设计评价

优点：

1. 字段直观，和 SRS 数据需求一致。
2. 简单结构适合课程项目。
3. 便于保存和展示。

问题：

1. 缺少封装，字段可被直接修改。
2. 业务行为集中在服务类，领域对象偏贫血模型。

建议：

1. 当前保持简单结构。
2. 若业务复杂化，可在领域对象中增加只读方法或状态判断方法。

## 7. 数据文件和构建脚本

### 7.1 设计评价

优点：

1. 文本文件便于课程演示和检查。
2. 构建脚本降低运行门槛。
3. Qt 运行库复制逻辑已包含在脚本中。

问题：

1. 文本文件不适合复杂查询和事务。
2. 构建脚本依赖本机固定 Qt 路径。

建议：

1. 后续迁移 SQLite。
2. 构建脚本改用环境变量或 CMake Presets。

## 8. 综合结论

当前模块设计满足课程项目现阶段需求，设计重点是“清晰、可演示、可维护”。从设计原则角度看，最值得改进的是：

1. 拆分 `DashboardWindow`。
2. 拆分 `RecruitmentService` 的存储职责。
3. 引入服务和仓储接口。
4. 增加服务层测试。

这些改进应在后续迭代逐步实施，不建议在提交阶段一次性大规模改动。
