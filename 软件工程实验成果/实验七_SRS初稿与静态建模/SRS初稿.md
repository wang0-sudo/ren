# 人才招聘系统软件需求规格说明 SRS 初稿

## 1. 引言

### 1.1 编写目的

本文档用于描述人才招聘系统的软件需求，明确系统功能、数据对象、接口、非功能需求和验收标准。本文档面向开发人员、测试人员、课程助教和项目审阅人员，是后续系统设计、编码实现、测试验证和答辩说明的重要依据。

### 1.2 项目背景

人才招聘系统是一个基于 `C++ + Qt Widgets` 的桌面端招聘管理系统。系统面向个人用户、企业用户和管理员三类角色，支持用户注册登录、会员审核、简历维护、岗位发布、岗位查询、岗位申请、留言处理等业务流程。

项目性质为软件工程课程项目，目标是在有限周期内完成一个结构清晰、功能闭环、可演示、可维护的轻量级系统。

## 2. 总体描述

### 2.1 产品前景

本系统定位为单机桌面招聘管理系统，用于模拟求职者、企业和管理员之间的招聘业务流程。系统当前不面向互联网部署，不提供多终端同步和在线支付等复杂功能。

### 2.2 产品功能概述

系统主要功能包括：

1. 用户注册与登录。
2. 管理员审核个人和企业用户。
3. 个人用户维护资料和简历。
4. 企业用户维护企业资料和岗位。
5. 个人用户查询岗位、申请岗位和撤销申请。
6. 企业用户查询已审核人才。
7. 用户留言，管理员回复或删除留言。
8. 本地数据保存和加载。

### 2.3 用户特征

| 用户 | 特征 | 使用重点 |
|---|---|---|
| 个人用户 | 具备基本计算机操作能力，关注岗位信息和简历投递 | 简历维护、岗位查询、申请管理 |
| 企业用户 | 代表企业发布招聘信息，关注人才筛选 | 企业资料、岗位管理、人才查询 |
| 管理员 | 负责平台准入和留言处理 | 用户审核、留言管理、系统维护 |

### 2.4 运行环境

| 项目 | 要求 |
|---|---|
| 操作系统 | Windows |
| 开发语言 | C++ |
| GUI 框架 | Qt Widgets |
| 编译环境 | Qt 6.x + MinGW |
| 数据存储 | 本地文本文件 `招聘数据.txt` |

### 2.5 设计约束

1. 系统采用桌面 GUI，不实现 Web 服务端。
2. 当前阶段使用本地文本文件存储数据。
3. 系统以课程演示为目标，使用模拟数据。
4. 业务规则应集中在 `RecruitmentService`，界面层负责交互。
5. 不同角色只能访问对应功能。

### 2.6 假设和依赖

1. 系统运行在单用户本地环境。
2. 用户输入的数据规模较小。
3. 管理员账号初始存在。
4. 文件系统可正常读写数据文件。
5. 演示数据不包含真实敏感个人信息。

## 3. 功能需求

### 3.1 用户注册与登录

| 编号 | 需求描述 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-LOGIN-01 | 系统应支持个人用户注册。 | 高 | 输入合法信息后生成待审核个人用户。 |
| FR-LOGIN-02 | 系统应支持企业用户注册。 | 高 | 输入合法信息后生成待审核企业用户。 |
| FR-LOGIN-03 | 系统应拒绝重复用户名。 | 高 | 重复用户名注册时给出错误提示。 |
| FR-LOGIN-04 | 系统应支持个人用户登录。 | 高 | 用户名和密码正确时进入个人用户界面。 |
| FR-LOGIN-05 | 系统应支持企业用户登录。 | 高 | 用户名和密码正确时进入企业用户界面。 |
| FR-LOGIN-06 | 系统应支持管理员登录。 | 高 | 管理员凭正确账号密码进入管理员界面。 |
| FR-LOGIN-07 | 系统应拒绝错误密码。 | 高 | 密码错误时不进入系统并提示。 |

### 3.2 个人用户功能

| 编号 | 需求描述 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-P-01 | 个人用户应能查看个人资料摘要。 | 高 | 个人主页显示姓名、联系方式、审核状态等。 |
| FR-P-02 | 个人用户应能修改个人基本信息。 | 高 | 保存后界面和数据文件更新。 |
| FR-P-03 | 个人用户应能维护简历。 | 高 | 教育、经历、技能、自我介绍可编辑保存。 |
| FR-P-04 | 个人用户应能查询可见岗位。 | 高 | 只显示启用且企业已审核通过的岗位。 |
| FR-P-05 | 个人用户应能申请岗位。 | 高 | 申请成功后加入已申请岗位列表。 |
| FR-P-06 | 系统应防止重复申请同一岗位。 | 高 | 重复申请时给出错误提示。 |
| FR-P-07 | 个人用户应能撤销岗位申请。 | 中 | 撤销后该岗位从已申请列表中移除。 |
| FR-P-08 | 个人用户应能修改密码。 | 中 | 旧密码正确时新密码保存。 |
| FR-P-09 | 个人用户应能提交留言。 | 中 | 管理员留言列表出现新留言。 |

### 3.3 企业用户功能

| 编号 | 需求描述 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-E-01 | 企业用户应能查看企业资料摘要。 | 高 | 企业主页显示企业名、联系人、审核状态等。 |
| FR-E-02 | 企业用户应能修改企业资料。 | 高 | 保存后界面和数据文件更新。 |
| FR-E-03 | 已审核企业应能发布岗位。 | 高 | 岗位保存后出现在企业岗位列表。 |
| FR-E-04 | 企业用户应能修改自己发布的岗位。 | 高 | 修改后岗位信息更新。 |
| FR-E-05 | 企业用户应能下架自己发布的岗位。 | 中 | 下架后个人用户不可见。 |
| FR-E-06 | 企业用户应能查询已审核人才。 | 中 | 人才列表显示已审核个人用户和简历摘要。 |
| FR-E-07 | 企业用户应能修改密码。 | 中 | 旧密码正确时新密码保存。 |
| FR-E-08 | 企业用户应能提交留言。 | 中 | 管理员留言列表出现新留言。 |

### 3.4 管理员功能

| 编号 | 需求描述 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-A-01 | 管理员应能查看待审核个人用户。 | 高 | 待审核个人用户显示在列表中。 |
| FR-A-02 | 管理员应能通过或驳回个人用户。 | 高 | 审核状态保存并刷新。 |
| FR-A-03 | 管理员应能查看待审核企业用户。 | 高 | 待审核企业用户显示在列表中。 |
| FR-A-04 | 管理员应能通过或驳回企业用户。 | 高 | 审核状态保存并刷新。 |
| FR-A-05 | 管理员应能查看留言。 | 中 | 留言列表显示来源、内容、状态和回复。 |
| FR-A-06 | 管理员应能回复留言。 | 中 | 回复后留言状态变为已处理。 |
| FR-A-07 | 管理员应能删除留言。 | 中 | 删除后留言不再显示。 |
| FR-A-08 | 管理员应能修改密码。 | 中 | 旧密码正确时新密码保存。 |

### 3.5 数据保存和加载

| 编号 | 需求描述 | 优先级 | 验收标准 |
|---|---|---|---|
| FR-D-01 | 系统启动时应加载已有数据。 | 高 | 数据文件存在时读取用户、岗位、留言。 |
| FR-D-02 | 系统无数据文件时应生成样例数据。 | 中 | 首次运行可进入演示状态。 |
| FR-D-03 | 系统变更数据后应保存到本地文件。 | 高 | 重启后数据仍存在。 |
| FR-D-04 | 系统保存失败时应返回错误信息。 | 中 | 界面或调用方可以获得失败提示。 |

## 4. 数据需求

### 4.1 数据对象

| 对象 | 字段 |
|---|---|
| PersonalUser | id、username、password、name、gender、phone、email、resume、appliedPositionIds、status |
| Resume | education、experience、skills、selfIntroduction |
| EnterpriseUser | id、username、password、companyName、contactPerson、phone、email、address、introduction、status |
| Position | id、enterpriseId、title、location、salary、requirement、description、active |
| Message | id、senderType、senderName、content、handled、reply |

### 4.2 数据关系

1. 一个个人用户拥有一份简历。
2. 一个企业用户可以发布多个岗位。
3. 一个个人用户可以申请多个岗位。
4. 一个岗位可以被多个个人用户申请。
5. 个人用户和企业用户均可以提交留言。
6. 管理员负责审核用户和处理留言。

## 5. 外部接口需求

### 5.1 用户界面

系统提供以下界面：

1. 登录窗口：角色选择、用户名、密码、登录、清空、个人注册、企业注册。
2. 个人用户工作台：个人资料、职位浏览、已申请岗位。
3. 企业用户工作台：企业信息、岗位管理、人才查询。
4. 管理员工作台：待审个人用户、待审企业用户、留言管理。

### 5.2 文件接口

系统使用 `招聘数据.txt` 保存数据。文件由 `RecruitmentService` 统一读写，界面层不直接解析文件。

### 5.3 操作系统接口

系统依赖 Windows 桌面环境和 Qt/MinGW 运行库。构建和运行脚本位于 `Qt图形界面/`。

## 6. 非功能需求

| 编号 | 类型 | 需求描述 | 验收方式 |
|---|---|---|---|
| NFR-01 | 易用性 | 界面按钮和表格应清晰，错误操作应提示。 | 人工检查主要流程。 |
| NFR-02 | 可靠性 | 数据保存后重启仍可读取。 | 执行重启加载测试。 |
| NFR-03 | 安全性 | 角色之间应隔离功能入口。 | 使用不同角色登录验证。 |
| NFR-04 | 安全性 | 后续应支持密码哈希存储。 | 设计文档中保留改进需求。 |
| NFR-05 | 可维护性 | 业务规则集中在 `RecruitmentService`。 | 代码审阅。 |
| NFR-06 | 可移植性 | 在指定 Windows + Qt 环境可构建运行。 | 执行构建脚本。 |
| NFR-07 | 性能 | 课程规模数据下列表刷新应基本即时。 | 人工体验检查。 |

## 7. 约束和风险

| 编号 | 约束/风险 | 影响 | 应对 |
|---|---|---|---|
| C-01 | 当前使用文本文件存储 | 数据一致性和安全性有限 | 后续可迁移 SQLite |
| C-02 | 密码当前存在明文风险 | 安全性不足 | 后续实现哈希 |
| C-03 | 课程周期有限 | 扩展功能难以全部完成 | 核心功能优先 |
| C-04 | 单机桌面系统 | 不支持多人在线协作使用 | 明确系统边界 |

## 8. 验收标准

| 编号 | 验收标准 |
|---|---|
| AC-01 | 个人用户和企业用户可以注册，并进入待审核状态。 |
| AC-02 | 管理员可以审核个人和企业用户。 |
| AC-03 | 个人用户可以维护简历、查询岗位、申请岗位和撤销申请。 |
| AC-04 | 企业用户可以维护企业资料、发布岗位、修改岗位和下架岗位。 |
| AC-05 | 企业用户可以查询已审核人才。 |
| AC-06 | 个人和企业用户可以留言，管理员可以回复或删除留言。 |
| AC-07 | 关键数据可以保存并在重启后加载。 |
| AC-08 | 文档、模型和代码描述一致。 |
