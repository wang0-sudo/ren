# SRS 模板对比与取舍

## 1. 对比目的

实验七要求搜集 SRS 编写案例，并对比《掌握需求过程》第 3 版附录 A 与国标 SRS 模板的差异。本文件从课程项目角度总结两类模板特点，并说明人才招聘系统 SRS 初稿采用的结构。

## 2. 国标 SRS 模板特点

国标 SRS 模板通常强调规范性、完整性和可审查性，适合正式软件项目归档。

| 特点 | 说明 |
|---|---|
| 章节结构固定 | 通常包含引言、总体描述、具体需求、接口、性能、属性等。 |
| 文档边界清楚 | 与可行性分析、设计说明、测试文档关系明确。 |
| 便于评审验收 | 需求编号、约束、验收标准有利于审查。 |
| 偏工程管理 | 更适合正式交付和阶段评审。 |

## 3. 《掌握需求过程》模板特点

《掌握需求过程》更强调从业务问题、利益相关者、使用场景和需求发现过程出发。

| 特点 | 说明 |
|---|---|
| 强调业务目标 | 先说明项目目的、业务事件和利益相关者。 |
| 强调场景和用例 | 使用场景帮助发现真实需求。 |
| 强调需求来源 | 关注需求如何从用户沟通和业务规则中得出。 |
| 强调可追踪性 | 需求需要与业务目标、验收条件、约束关联。 |

## 4. 两类模板对比

| 维度 | 国标 SRS 模板 | 《掌握需求过程》模板 | 对本项目启发 |
|---|---|---|---|
| 文档风格 | 规范、正式 | 面向需求发现和沟通 | 课程提交采用规范结构，同时保留场景描述。 |
| 需求来源 | 较少展开获取过程 | 重视利益相关者和业务事件 | 增加用户角色和业务流程说明。 |
| 功能描述 | 适合编号和验收 | 适合从场景推导功能 | 功能需求使用编号，关键流程使用场景补充。 |
| 非功能需求 | 作为独立章节 | 与质量约束和验收关联 | 保留非功能章节并写明验收方式。 |
| 适用场景 | 正式项目文档 | 需求分析和用户沟通 | 本项目两者结合。 |

## 5. 人才招聘系统 SRS 取舍

本项目 SRS 采用“国标结构为主，场景和验收补充”的方式。

保留内容：

1. 引言。
2. 总体描述。
3. 功能需求。
4. 数据需求。
5. 外部接口需求。
6. 非功能需求。
7. 约束和风险。
8. 验收标准。
9. 需求跟踪矩阵。

补充内容：

1. 三类用户特征。
2. 核心业务流程。
3. 需求优先级。
4. 每条功能需求的验收标准。
5. 与当前代码模块的映射。

## 6. 结论

国标模板更适合作为课程提交文档骨架，《掌握需求过程》的思想更适合补充需求来源、用户场景和验收条件。人才招聘系统 SRS 初稿采用二者结合的方式：结构上保持正式，内容上强调用户角色、核心流程、可验证需求和需求跟踪。
