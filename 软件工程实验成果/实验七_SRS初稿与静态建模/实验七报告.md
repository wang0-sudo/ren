## 2. SRS 案例和模板学习

### 2.1 SRS 编写案例总结

通过对常见 SRS 示例和实验六 SRS 结构的整理，可以得到以下写作原则：

1. 需求必须可验证，避免“界面友好”“功能完善”等不可验收表述。
2. 每条功能需求应有编号、优先级和验收标准。
3. 需求应区分功能需求、数据需求、接口需求和非功能需求。
4. 用户角色和业务流程应在总体描述中说明。
5. SRS 不应写过多设计实现细节，但可以记录重要约束。

### 2.2 模板对比结论

国标 SRS 模板偏正式、结构化，适合课程提交和评审；《掌握需求过程》的模板更重视业务目标、利益相关者、需求来源和场景。人才招聘系统采用两者结合方式：文档结构参考国标，需求描述补充用户角色、业务场景、优先级和验收标准。

详细对比见 `SRS模板对比.md`。

## 3. SRS 初稿内容

本次 SRS 初稿包括以下章节：

1. 引言。
2. 总体描述。
3. 功能需求。
4. 数据需求。
5. 外部接口需求。
6. 非功能需求。
7. 约束和风险。
8. 验收标准。
9. 需求跟踪矩阵。

其中功能需求按角色划分：

| 角色 | 需求范围 |
|---|---|
| 个人用户 | 注册、登录、资料维护、简历维护、岗位查询、岗位申请、撤销申请、留言、修改密码 |
| 企业用户 | 注册、登录、企业资料维护、岗位发布、岗位修改、岗位下架、人才查询、留言、修改密码 |
| 管理员 | 登录、审核个人用户、审核企业用户、查看留言、回复留言、删除留言、修改密码 |
| 系统 | 数据加载、样例数据生成、数据保存、保存失败处理 |

SRS 初稿见 `SRS初稿.md`。

## 4. 静态建模

### 4.1 用例模型

用例模型描述三类用户与系统功能之间的关系。个人用户关注岗位和简历，企业用户关注岗位和人才，管理员关注审核和留言。

### 4.2 领域类图

领域类图直接对应当前代码结构：

| 类/结构 | 说明 |
|---|---|
| `RecruitmentService` | 业务服务层，负责注册、登录、岗位、审核、留言和保存加载 |
| `PersonalUser` | 个人用户 |
| `Resume` | 简历 |
| `EnterpriseUser` | 企业用户 |
| `Position` | 岗位 |
| `Message` | 留言 |
| `AuditStatus` | 审核状态 |

### 4.3 E-R 模型

E-R 模型描述数据对象之间的关系。虽然当前系统使用文本文件存储，但 E-R 模型可作为后续迁移 SQLite 的基础。

### 4.4 模块结构图

模块结构图将系统划分为界面层、业务层和存储层：

1. 界面层：`LoginWindow`、`DashboardWindow`。
2. 业务层：`RecruitmentService`。
3. 存储层：`招聘数据.txt`。

完整静态模型见 `静态模型.md`。

## 5. 与用户沟通后的需求归纳

根据当前项目目标和已有原型，需求归纳为以下核心结论：

1. 系统以三类角色为中心，角色边界必须清晰。
2. 注册后的个人和企业都需要经过管理员审核。
3. 岗位只应由已审核企业发布，并只对个人用户显示有效岗位。
4. 个人用户的岗位申请应支持查看和撤销。
5. 企业用户能查看已审核人才，但不应直接修改个人用户信息。
6. 留言功能作为用户与平台沟通渠道，由管理员统一处理。
7. 数据保存和重启加载是演示必须满足的基础能力。
