# 人才招聘系统软件需求规格说明 SRS 定稿

## 1. 引言

### 1.1 编写目的

本文档定义人才招聘系统的软件需求，作为体系结构设计、详细设计、编码实现、测试验证和课程答辩的需求基线。本文档整合实验七 SRS 初稿、实验八动态模型和实验九架构设计前置分析。

### 1.2 项目背景

人才招聘系统是一个基于 `C++ + Qt Widgets` 的 Windows 桌面端招聘管理系统。系统服务于个人用户、企业用户和管理员三类角色，支持用户注册登录、管理员审核、简历维护、岗位发布、岗位查询、岗位申请、留言处理和本地数据持久化。

### 1.3 术语

| 术语 | 说明 |
|---|---|
| SRS | 软件需求规格说明。 |
| SAD | 软件体系结构设计文档。 |
| 个人用户 | 求职者角色，可维护资料和简历，可查询并申请岗位。 |
| 企业用户 | 招聘方角色，可维护企业资料，可发布和管理岗位。 |
| 管理员 | 平台管理角色，可审核用户并处理留言。 |
| 审核状态 | `Pending`、`Approved`、`Rejected`。 |
| 可见岗位 | 启用且发布企业已审核通过的岗位。 |

## 2. 总体描述

### 2.1 产品定位

本系统定位为课程演示级单机桌面招聘管理系统，用于模拟招聘平台中个人用户、企业用户和管理员之间的核心业务流程。系统不提供 Web 访问、在线支付、邮件通知和多人实时协作。

### 2.2 用户类别

| 用户 | 主要目标 |
|---|---|
| 个人用户 | 注册登录、维护简历、查询岗位、申请岗位、撤销申请、提交留言。 |
| 企业用户 | 注册登录、维护企业资料、发布岗位、修改岗位、下架岗位、查询人才、提交留言。 |
| 管理员 | 登录后台、审核个人用户、审核企业用户、回复留言、删除留言、修改密码。 |

### 2.3 运行环境

| 项目 | 要求 |
|---|---|
| 操作系统 | Windows |
| 开发语言 | C++17 |
| GUI 框架 | Qt Widgets |
| 构建环境 | Qt 6.x + MinGW / CMake |
| 数据存储 | 本地文本文件 `招聘数据.txt` |

### 2.4 设计约束

1. 系统采用桌面 GUI。
2. 当前阶段使用文本文件保存数据。
3. 业务规则集中在 `RecruitmentService`。
4. 表示层不得直接解析数据文件。
5. 三类角色只能访问本角色对应功能。
6. 项目以课程演示为主要目标。

## 3. 功能需求

|---|---|---|---|

| FR-LOGIN-02 | 系统应支持企业用户注册。 | 高 | 输入合法信息后生成待审核企业用户。 |
| FR-LOGIN-03 | 系统应拒绝重复用户名。 | 高 | 重复用户名注册失败并提示。 |
| FR-LOGIN-04 | 系统应支持个人用户登录。 | 高 | 用户名和密码正确时进入个人工作台。 |
| FR-LOGIN-05 | 系统应支持企业用户登录。 | 高 | 用户名和密码正确时进入企业工作台。 |

## 4. 数据需求

| 对象 | 字段 |
|---|---|
| `PersonalUser` | `id`、`username`、`password`、`name`、`gender`、`phone`、`email`、`resume`、`appliedPositionIds`、`status` |
| `Resume` | `education`、`experience`、`skills`、`selfIntroduction` |
| `EnterpriseUser` | `id`、`username`、`password`、`companyName`、`contactPerson`、`phone`、`email`、`address`、`introduction`、`status` |
| `Position` | `id`、`enterpriseId`、`title`、`location`、`salary`、`requirement`、`description`、`active` |
| `Message` | `id`、`senderType`、`senderName`、`content`、`handled`、`reply` |

数据关系：

1. 一个个人用户拥有一份简历。
2. 一个企业用户可以发布多个岗位。
3. 一个个人用户可以申请多个岗位。
4. 一个岗位可以被多个个人用户申请。
5. 个人用户和企业用户可以提交多条留言。

## 5. 动态行为需求

### 5.1 用户审核生命周期

```text
NotRegistered -> Pending -> Approved
NotRegistered -> Pending -> Rejected
```

个人用户审核通过后可以申请岗位；企业用户审核通过后可以发布岗位。

### 5.2 岗位生命周期

```text
Created -> Active -> Visible
Active/Visible -> Inactive
```

岗位可见条件：

```text
position.active = true
and enterprise.status = Approved
```

### 5.3 岗位申请生命周期

```text
NotApplied -> Applied -> Withdrawn
```

岗位申请必须满足个人审核通过、岗位可见、未重复申请。

### 5.4 留言生命周期

```text
Draft -> Pending -> Handled
Pending/Handled -> Deleted
```

## 6. 业务约束

| 编号 | 约束 |
|---|---|
| BR-01 | 用户名在个人用户、企业用户和管理员之间唯一。 |
| BR-02 | 个人和企业用户注册后默认进入 `Pending`。 |
| BR-03 | 未审核通过的个人用户不能申请岗位。 |
| BR-04 | 未审核通过的企业用户不能发布岗位。 |
| BR-05 | 可见岗位必须处于启用状态且发布企业已审核通过。 |
| BR-06 | 同一个人用户不能重复申请同一岗位。 |
| BR-07 | 企业用户只能修改或下架自己发布的启用岗位。 |
| BR-08 | 留言内容不能为空。 |
| BR-09 | 留言回复内容不能为空。 |
| BR-10 | 业务数据变更后必须保存。 |

## 7. 外部接口需求

### 7.1 用户界面

系统提供登录窗口和三类角色工作台：

1. 登录窗口：角色选择、用户名、密码、登录、注册入口。
2. 个人工作台：个人资料、简历、职位浏览、已申请岗位、留言。
3. 企业工作台：企业信息、岗位管理、人才查询、留言。
4. 管理员工作台：个人审核、企业审核、留言处理、密码修改。

### 7.2 文件接口

系统使用 `招聘数据.txt` 保存数据，文件由 `RecruitmentService` 统一读写。
