# SRS 补充与项目跟踪更新

## 1. 需要并入 SRS 的动态行为章节

建议在实验七 `SRS初稿.md` 的功能需求和数据需求之后增加“动态行为模型”章节，内容如下。

### 1.1 用户审核生命周期

个人用户和企业用户注册后均进入待审核状态。管理员可以将待审核用户审核为通过或驳回。审核通过的个人用户可以申请岗位，审核通过的企业用户可以发布岗位。审核驳回的用户仍可登录查看状态，但不能进入需要审核通过的业务流程。

状态集合：

```text
AuditStatus = { Pending, Approved, Rejected }
```

状态迁移：

```text
NotRegistered -> Pending -> Approved
NotRegistered -> Pending -> Rejected
```

### 1.2 岗位生命周期

企业用户发布岗位后生成 `Position` 对象，默认 `active = true`。岗位是否对个人用户可见，不仅取决于岗位自身启用状态，也取决于发布企业的审核状态。

岗位可见条件：

```text
position.active = true
and enterpriseById(position.enterpriseId).status = Approved
```

岗位下架后：

```text
position.active = false
```

下架岗位不再出现在个人用户可申请岗位列表中。

### 1.3 岗位申请生命周期

个人用户申请岗位的状态变化如下：

```text
未申请 -> 已申请 -> 已撤销
```

申请成功条件：

1. 个人用户存在。
2. 个人用户审核状态为 `Approved`。
3. 岗位存在且可见。
4. 个人用户尚未申请过该岗位。

撤销成功条件：

1. 个人用户存在。
2. `appliedPositionIds` 中存在目标岗位编号。

### 1.4 留言处理生命周期

留言由个人用户或企业用户提交，管理员负责处理。新留言初始为未处理状态；管理员回复后进入已处理状态；管理员也可以删除留言。

状态变化：

```text
未提交 -> 未处理 -> 已处理
未处理 -> 已删除
已处理 -> 已删除
```

## 2. 业务约束补充

| 编号 | 约束 | 说明 |
|---|---|---|
| BR-01 | 用户名在个人用户、企业用户和管理员之间唯一 | 防止登录身份冲突。 |
| BR-02 | 个人用户注册后默认 `Pending` | 必须经过管理员审核。 |
| BR-03 | 企业用户注册后默认 `Pending` | 必须经过管理员审核。 |
| BR-04 | 未审核通过的个人用户不能申请岗位 | 对应 `applyForPosition()` 的审核状态校验。 |
| BR-05 | 未审核通过的企业用户不能发布岗位 | 对应 `addPosition()` 的审核状态校验。 |
| BR-06 | 可见岗位必须处于启用状态且发布企业已通过审核 | 对应 `isPositionVisible()`。 |
| BR-07 | 同一个人用户不能重复申请同一岗位 | 对应 `appliedPositionIds` 去重。 |
| BR-08 | 企业用户只能修改或下架自己发布的启用岗位 | 防止越权操作。 |
| BR-09 | 留言内容不能为空 | 对应 `addMessage()` 输入校验。 |
| BR-10 | 留言回复内容不能为空 | 对应 `replyMessage()` 输入校验。 |
| BR-11 | 所有业务数据变更后必须保存到本地数据文件 | 保障重启后数据仍可读取。 |

## 3. 异常流程补充

| 编号 | 异常场景 | 系统响应 |
|---|---|---|
| EX-01 | 注册时用户名为空 | 拒绝注册并提示用户名不能为空。 |
| EX-02 | 注册时密码为空 | 拒绝注册并提示密码不能为空。 |
| EX-03 | 注册时用户名重复 | 拒绝注册并提示更换用户名。 |
| EX-04 | 登录时密码错误 | 登录失败，不进入工作台。 |
| EX-05 | 未审核个人用户申请岗位 | 拒绝申请并提示账号未审核通过。 |
| EX-06 | 申请不存在或不可见岗位 | 拒绝申请并提示岗位不可申请。 |
| EX-07 | 重复申请同一岗位 | 拒绝申请并提示已申请过该岗位。 |
| EX-08 | 企业发布岗位时标题为空 | 拒绝发布并提示岗位名称不能为空。 |
| EX-09 | 企业修改非本人岗位 | 拒绝修改并提示未找到该岗位。 |
| EX-10 | 留言内容为空 | 拒绝提交并提示留言内容不能为空。 |
| EX-11 | 回复内容为空 | 拒绝回复并提示回复内容不能为空。 |
| EX-12 | 保存数据文件失败 | 返回保存失败信息，调用方提示用户。 |

## 6. 对后续实验的建议

1. 实验九进行 SAD 编写时，可将 `RecruitmentService` 定义为业务服务层核心组件，将 `LoginWindow` 和 `DashboardWindow` 定义为表示层组件。
2. 软件测试设计应优先覆盖审核准入、岗位可见性、重复申请、留言处理和数据持久化。
3. 如果后续扩展数据库存储，当前 ER 模型、数据流模型和业务约束可以直接作为表设计与事务设计依据。
