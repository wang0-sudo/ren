# 实验八报告：动态建模与 Petri 网建模

## 1. 实验目的

本次实验面向人才招聘系统，在实验七 SRS 初稿和静态模型基础上补充动态模型。实验目标如下：

1. 学习 Petri 网的基本概念、组成元素和分析方法。
2. 使用状态图、活动图、数据流图、Petri 网和逻辑约束描述系统动态行为。
3. 将动态建模结果补充到 SRS 中，使需求描述不仅包含对象结构，也包含业务流程、状态变化和约束条件。
4. 更新项目跟踪表，为后续软件体系结构设计和测试设计提供依据。

## 2. Petri 网学习总结

Petri 网适合描述存在并发、同步、互斥和资源约束的软件系统过程。其基本组成包括：

| 元素 | 含义 |
|---|---|
| Place | 库所，表示系统条件、状态或资源。 |
| Transition | 变迁，表示事件、动作或业务操作。 |
| Arc | 有向弧，连接库所和变迁，说明输入与输出关系。 |
| Token | 标识，表示某个库所当前持有的状态或资源。 |
| Marking | 标识分布，表示系统某一时刻的整体状态。 |

Petri 网的基本运行规则是：当一个变迁的所有输入库所都具有足够标识时，该变迁处于可发生状态；变迁发生后，从输入库所移除标识，并向输出库所添加标识。该规则可以自然表达系统中的前置条件、流程推进和状态改变。

在人才招聘系统中，个人注册、企业注册、管理员审核、岗位发布、岗位申请和留言处理之间存在明显的先后关系和同步条件。例如，个人用户必须审核通过后才能申请岗位；企业用户必须审核通过后才能发布可见岗位；岗位申请需要同时满足“个人已通过审核”和“岗位可见”两个条件。因此 Petri 网适合作为本系统动态建模的补充工具。

## 3. 动态建模对象

本次建模围绕当前代码和 SRS 中已经确定的核心对象展开：

| 对象 | 动态关注点 |
|---|---|
| PersonalUser | 注册后进入待审核状态，经管理员审核后变为通过或驳回；通过后可维护简历、申请岗位和撤销申请。 |
| EnterpriseUser | 注册后进入待审核状态，经管理员审核后变为通过或驳回；通过后可发布、修改和下架岗位。 |
| Position | 创建后默认启用，启用且企业已通过审核时对个人用户可见；下架后不再可见。 |
| Message | 用户提交后进入未处理状态，管理员回复后进入已处理状态，也可以被删除。 |
| RecruitmentService | 作为业务规则中心，负责校验状态、维护数据集合并持久化到 `招聘数据.txt`。 |

## 4. 建模成果

### 4.1 状态模型

`动态模型.md` 中补充了个人用户审核状态、企业用户审核状态、岗位生命周期和留言处理状态。状态模型明确了以下规则：

1. 注册产生的个人用户和企业用户初始状态均为 `Pending`。
2. 管理员审核后，用户状态只能进入 `Approved` 或 `Rejected`。
3. 企业用户发布岗位时必须处于 `Approved` 状态。
4. 岗位下架后不再出现在个人用户可申请列表中。
5. 留言回复后设置 `handled = true`，删除后退出当前业务处理流程。

### 4.2 活动模型

活动模型重点描述个人用户申请岗位的业务流程：

1. 个人用户登录。
2. 系统检查个人审核状态。
3. 系统读取可见岗位列表。
4. 用户选择岗位并提交申请。
5. 系统检查岗位是否可见、是否重复申请。
6. 申请成功后写入 `appliedPositionIds` 并持久化。

该活动流程直接对应 `RecruitmentService::applyForPosition()` 的业务校验顺序，可用于后续测试用例设计。

### 4.3 数据流模型

数据流图将系统划分为外部实体、处理过程和数据存储：

| 类型 | 内容 |
|---|---|
| 外部实体 | 个人用户、企业用户、管理员 |
| 处理过程 | 注册登录、审核管理、岗位管理、申请管理、留言管理 |
| 数据存储 | 个人用户数据、企业用户数据、岗位数据、留言数据、本地数据文件 |

数据流模型说明了界面层不直接操作文件，所有数据读写由 `RecruitmentService` 统一完成。

### 4.4 Petri 网模型

`Petri网模型.md` 中建立了人才招聘系统核心业务 Petri 网。该模型包含个人审核、企业审核、岗位发布、岗位申请和留言处理五类子流程，并分析了：

1. 并发性：个人注册审核与企业注册审核可以独立发生。
2. 同步性：岗位申请需要同时满足个人已审核通过和岗位可见。
3. 冲突性：待审核用户在通过和驳回之间只能选择一个结果。
4. 有界性：单个用户审核状态在任一时刻只能处于一种状态。
5. 活性：只要管理员执行审核，待审核流程可以继续推进。

## 5. 对 SRS 的补充

本次实验对 SRS 的补充包括：

1. 增加“动态行为模型”章节，说明核心对象状态和业务流程。
2. 增加“业务约束”章节，记录唯一用户名、审核准入、岗位可见性、申请去重和留言处理约束。
3. 增加“异常流程”说明，例如未审核用户申请岗位、重复申请、下架岗位申请、空留言回复等。
4. 将动态模型映射到需求编号，便于后续设计和测试追踪。

## 7. 实验结论

通过本次实验，人才招聘系统的需求模型从静态对象结构扩展到动态业务行为。状态图明确了用户、岗位和留言对象的生命周期；活动图和数据流图说明了关键业务流程；Petri 网进一步表达了系统中的并发、同步和冲突关系。这些成果可以直接支撑后续软件体系结构设计、详细设计和测试用例设计。
