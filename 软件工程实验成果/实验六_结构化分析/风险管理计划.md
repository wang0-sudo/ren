# 人才招聘系统风险管理细化

## 1. 风险管理目标

风险管理的目标不是消除所有不确定性，而是尽早识别高影响风险，制定预防措施和应急预案，避免项目在交付前集中暴露问题。

## 2. 风险分级规则

概率 P 和影响 I 取值 1 到 5：

`R = P * I`

| 风险值 | 等级 | 管理动作 |
|---:|---|---|
| 1-4 | 低 | 记录，阶段性复查 |
| 5-9 | 中 | 指定预防措施 |
| 10-16 | 高 | 每周跟踪，准备应急预案 |
| 17-25 | 很高 | 优先处理，必要时调整范围或计划 |

## 4. 风险优先级排序

| 优先级 | 风险 | 处理顺序理由 |
|---:|---|---|
| 1 | R6 主流程缺少回归测试 | 直接影响演示成败 |
| 2 | R1 功能范围持续扩大 | 直接影响进度和文档质量 |
| 3 | R2 文档和代码任务堆积 | 影响后续实验连续交付 |
| 4 | R5 角色越权 | 影响系统正确性和安全性 |
| 5 | R4 密码明文保存 | 安全风险高，但课程演示可先用模拟数据控制 |
| 6 | R3 数据格式失败 | 影响演示稳定性 |

## 5. 风险应对计划

### 5.1 需求风险

控制范围：

1. 核心功能：注册登录、审核、简历、岗位、申请、留言。
2. 重要功能：密码修改、搜索、数据保存。
3. 扩展功能：推荐、通知、统计图、数据库迁移。

处理原则：核心功能必须完成，重要功能尽量完成，扩展功能只在时间允许时实现。

### 5.2 进度风险

每周检查：

1. 是否有未审阅实验。
2. 是否有未推送成果。
3. 是否有关键路径任务延期。
4. 是否有文档需要返工。

### 5.3 技术风险

技术预案：

1. 文本存储保持稳定格式。
2. 重要数据文件保留样例。
3. 服务层统一处理业务规则。
4. 后续设计中预留 SQLite 迁移方案。

### 5.4 安全风险

安全要求：

1. 不使用真实个人数据。
2. 文档明确密码哈希需求。
3. 不同角色只看到自己的功能入口。
4. 服务层补充越权校验。

### 5.5 质量风险

主流程测试清单：

| 编号 | 测试场景 | 期望结果 |
|---|---|---|
| T1 | 个人注册 | 生成待审核个人用户 |
| T2 | 企业注册 | 生成待审核企业用户 |
| T3 | 管理员审核 | 用户状态正确变更 |
| T4 | 个人维护简历 | 保存后可再次读取 |
| T5 | 企业发布岗位 | 岗位在列表中显示 |
| T6 | 个人申请岗位 | 申请记录保存 |
| T7 | 个人撤销申请 | 申请记录删除 |
| T8 | 用户留言 | 管理员可查看 |
| T9 | 管理员回复留言 | 留言状态变为已处理 |
| T10 | 重启程序 | 数据仍能加载 |

## 6. 风险复盘机制

1. 每个实验结束后更新风险状态。
2. 每次提交前检查 staged 文件，避免误提交。
3. 每次文档更新后核对是否与代码功能一致。
4. 每次演示前执行主流程测试清单。

## 7. 风险管理结论

当前项目风险以范围、进度、测试、安全和数据可靠性为主。通过核心功能优先、短周期审阅、测试清单、服务层校验和文档同步，可以将风险控制在课程项目可接受范围内。
