# 面向对象设计原则分析

## 1. 单一职责原则

### 1.1 原则说明

单一职责原则要求一个类或模块应该只有一个引起它变化的原因。类的职责越集中，修改影响越容易控制。

### 1.2 项目应用

当前项目已经有初步职责分离：

| 模块 | 职责 |
|---|---|
| `LoginWindow` | 登录和注册入口。 |
| `DashboardWindow` | 三类角色工作台展示与交互。 |
| `RecruitmentService` | 业务规则、数据集合和持久化。 |

这比将所有逻辑写在一个窗口中更清晰。

### 1.3 存在问题

`DashboardWindow` 同时包含个人用户、企业用户和管理员三类界面构建、数据刷新和事件处理。它可能因为以下原因变化：

1. 个人用户界面变化。
2. 企业用户界面变化。
3. 管理员审核界面变化。
4. 表格展示样式变化。
5. 业务交互方式变化。

`RecruitmentService` 也存在职责集中问题，它同时负责：

1. 业务规则。
2. 数据集合管理。
3. 数据文件读写。
4. 样例数据生成。

### 1.4 改进建议

1. 将 `DashboardWindow` 拆成 `PersonalPanel`、`EnterprisePanel`、`AdminPanel`。
2. 将 `RecruitmentService` 的文件读写拆到 `RecruitmentRepository`。
3. 保留 `RecruitmentService` 作为业务协调层。

## 2. 开闭原则

### 2.1 原则说明

开闭原则要求软件实体对扩展开放，对修改关闭。新增功能时应尽量通过增加新代码完成，而不是大量修改已有稳定代码。

### 2.2 项目应用

当前系统新增一个业务操作时，可以通过在 `RecruitmentService` 中增加公开方法，并在对应窗口事件中调用来扩展。例如新增“修改密码”功能时，可以增加 `changePersonalPassword()`、`changeEnterprisePassword()`、`changeAdminPassword()`。

### 2.3 存在问题

当前保存和加载逻辑直接依赖每个结构体字段。如果给 `Position` 增加 `category` 或 `publishTime` 字段，需要同时修改：

1. `Position` 结构体。
2. `save()` 写出逻辑。
3. `load()` 读取逻辑。
4. UI 表格展示。
5. SRS 和 SAD 文档。

### 2.4 改进建议

1. 抽取序列化/反序列化函数，减少 `load()` 和 `save()` 主流程变化。
2. 使用更结构化的存储方式，如 SQLite。
3. 引入仓储接口，让业务层对存储扩展更开放。

## 3. 里氏替换原则

### 3.1 原则说明

里氏替换原则要求子类对象应当能够替换父类对象，并保持程序正确性。该原则强调继承关系应符合行为契约。

### 3.2 项目应用

当前项目中业务层较少使用继承，主要继承来自 Qt：

1. `LoginWindow : QWidget`
2. `DashboardWindow : QMainWindow`

这类继承符合 Qt 框架的使用方式，窗口类可以被当作对应 Qt 基类使用。

### 3.3 存在问题

业务层缺少可替换接口，例如窗口类依赖具体的 `RecruitmentService`。如果后续想替换为测试服务、远程服务或模拟服务，需要修改窗口类构造函数类型。

### 3.4 改进建议

定义抽象服务接口：

```cpp
class IRecruitmentService {
public:
    virtual ~IRecruitmentService() = default;
    virtual bool registerPersonalUser(...) = 0;
    virtual PersonalUser* loginPersonal(...) = 0;
    virtual bool applyForPosition(...) = 0;
};
```

然后由 `RecruitmentService` 实现该接口。窗口依赖接口后，可用测试替身替换真实服务。

## 4. 迪米特法则

### 4.1 原则说明

迪米特法则要求对象尽量少了解其他对象的内部细节，只与直接朋友通信，以降低耦合。

### 4.2 项目应用

当前窗口类主要通过 `RecruitmentService` 访问业务数据，而不是直接读写 `招聘数据.txt`。这体现了迪米特法则：

```text
DashboardWindow -> RecruitmentService -> 招聘数据.txt
```

界面层不需要知道文件格式。

### 4.3 存在问题

窗口类仍直接读取领域结构体字段，例如个人姓名、审核状态、岗位标题、留言内容等。字段变化会影响窗口代码。

### 4.4 改进建议

1. 为界面层提供 DTO 或 ViewModel。
2. 让服务层返回适合表格展示的数据结构。
3. 将状态文本转换等展示辅助逻辑集中封装。

## 5. 依赖倒转原则

### 5.1 原则说明

依赖倒转原则要求高层模块不应依赖低层模块，二者都应依赖抽象；抽象不应依赖细节，细节应依赖抽象。

### 5.2 项目应用

当前项目已有构造函数注入：

```cpp
LoginWindow(std::shared_ptr<RecruitmentService> service, QWidget* parent = nullptr);
DashboardWindow(std::shared_ptr<RecruitmentService> service,
                UserRole role,
                int currentUserId,
                QWidget* parent = nullptr);
```

窗口不在内部创建服务对象，这比直接 `new RecruitmentService` 更符合依赖倒转思想。

### 5.3 存在问题

窗口依赖的仍是具体类 `RecruitmentService`，不是抽象接口。`RecruitmentService` 又直接依赖文本文件存储细节。

### 5.4 改进建议

1. 抽象 `IRecruitmentService`。
2. 抽象 `IRecruitmentRepository`。
3. 让具体类实现接口，由 `主程序.cpp` 或组合根负责创建和注入。

## 6. 合成复用原则

### 6.1 原则说明

合成复用原则强调优先使用组合和聚合复用已有功能，而不是滥用继承。

### 6.2 项目应用

当前项目大量使用组合：

1. `LoginWindow` 组合 `QComboBox`、`QLineEdit`、`QLabel` 和 `RecruitmentService`。
2. `DashboardWindow` 组合 `QTabWidget`、`QTableWidget`、输入框和服务对象。
3. `PersonalUser` 组合 `Resume`。
4. `RecruitmentService` 组合多个领域对象集合。

### 6.3 优点

1. 控件和业务对象可独立变化。
2. 不需要为每种业务扩展创建复杂继承层级。
3. 对课程项目而言结构更直接。

### 6.4 改进建议

继续优先使用组合。后续拆分界面时，可以让 `DashboardWindow` 组合 `PersonalPanel`、`EnterprisePanel`、`AdminPanel`，而不是通过继承创建多个特殊窗口。

## 7. 原则应用总结

| 原则 | 当前评分 | 说明 |
|---|---|---|
| 单一职责原则 | 中 | 已分层，但窗口和服务类职责仍较集中。 |
| 开闭原则 | 中 | 可扩展业务方法，但字段和存储扩展仍需修改多处。 |
| 里氏替换原则 | 中 | Qt 继承使用合理，业务接口抽象不足。 |
| 迪米特法则 | 中上 | UI 不直接接触文件，但仍接触领域字段细节。 |
| 依赖倒转原则 | 中 | 有构造函数注入，缺少抽象接口。 |
| 合成复用原则 | 高 | 项目主要通过组合组织窗口、控件和服务。 |
